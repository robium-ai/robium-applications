# 2026-07-14 — vla-trial, Task 5 (scripted IK oracle / day-1 canary)

## Figured out from scratch — the EE site is not the grasp point, and nothing says so

- **[lerobot] / [none] — a robot arm's "end-effector site" is NOT where the object is held, and no skill says to check.** The
  SO-101's `gripperframe` site sits at gripper-local `(0.012, ~0, -0.098)`, out near the fingertips. But the arm does not
  hold a 4x4x6 cm cube at its fingertips at all — it holds it in the **jaw throat**, at gripper-local `z ~= -0.035`, up near
  the jaw root. Aiming IK at the site (the obvious move, and what the task brief sketched) put the actual grasp ~7 cm from
  where I thought it was. Expected: solve IK to the EE site, close gripper, done. Happened: jaws closed on empty air every
  time, arm stalled, cube knocked over, 0/10.
  **How I found it (the generalisable technique):** brute-force calibration — park the arm, teleport the object to a grid of
  candidate *gripper-local* offsets, close the gripper, lift, keep the offsets whose object came up with the arm.
  ```
  HOLD  local=(-0.020, 0, -0.040)  grip=-0.172  z 0.145 -> 0.214
  HOLD  local=(-0.020, 0, -0.030)  grip=-0.175  z 0.149 -> 0.217
  ```
  This is 20 lines of code and would have saved the entire session. **A manipulation skill should tell you to calibrate the
  grasp offset empirically before writing a single waypoint**, rather than assuming the EE site/frame is the grasp point.
  Fix verified: the throat offset is the only reference that ever produces a hold.

- **[none] The decisive debugging move for "arm won't reach": delete the object and re-run.** I burned many iterations
  unsure whether the arm was torque-limited, IK-broken, or blocked. One control settled it:
  ```
  cube PRESENT:  arm_err=0.1835  saturated_joints=[1 2]  qfrc=[-1.21 2.94 2.94 1.66 -0.17]
  cube BANISHED: arm_err=0.0006  saturated_joints=[]     qfrc=[0. -0.51 -0.3 -0.11 -0.]
  ```
  Same commanded pose, object teleported out of the scene. Instantly rules out IK error, gravity, torque limits, and
  unreachability, and proves the object is the obstruction. **This belongs in a debugging playbook** — it is the
  manipulation analogue of bisecting.

- **[none] Position-only IK silently returns a wrist orientation you never chose, and it can be an unusable one.** The
  SO-101 can only reach its workspace with fingers ~32 deg BELOW horizontal. That orientation makes the fixed jaw's shaft
  sweep through the object. Constraining the finger axis to level is *genuinely unreachable* (`pos_err=0.094`,
  `ori_err=0.527`, and the pose stalls even with the object removed). So "just add an orientation constraint" is not always
  available on a 5-DOF arm — and position-only IK gives you no signal that the orientation it picked is a grasp-killer.

- **[none] Unnamed MJCF collision geoms are invisible to name-based contact checks.** The geom that actually blocked the
  grasp (`wrist_roll_follower_so101_gripper_part0_v1`, geom 31, the fixed jaw's collision *mesh*) has **no `name` attribute**.
  Our `GRIPPER_GEOMS` config is a hand-maintained *name* list, so `_is_released()` cannot see it. Harmless today (nothing is
  ever held) but a live false-success risk once the grasp works. **Resolve gripper geoms by `geom_bodyid` membership, not by
  a name list** — names are optional in MJCF and menagerie leaves mesh geoms unnamed.

## Worked as documented

- **[none] The brief's instruction to determine gripper polarity empirically was exactly right, and cheap.** Drive the
  gripper joint to each end of `actuator_ctrlrange`, measure the fingertip gap:
  `-0.17453 -> 0.0041 m (CLOSED)`, `+1.74533 -> 0.1334 m (OPEN)`. So ctrlrange LOW=closed, HIGH=open. Took 5 minutes and
  removed a whole class of doubt. (The upstream SO-ARM100 docs claim the opposite convention and admit their MJCF disagrees.)
  Polarity turned out **not** to be the bug — but knowing that for certain is what let me rule it out and keep digging.

## Noise / misleading guidance

- **[none] "Raise SETTLE_STEPS" is a debugging dead-end when the arm is *blocked* rather than *converging*.** The task brief
  lists "too few settle steps" as debug step 3. I swept 12 -> 20 -> 40 -> 60 steps per waypoint: **zero effect**, because the
  arm was stalled against a solid object with two joints saturated at their forcerange, not still settling. The tell that
  distinguishes the two: check `qfrc_actuator` against `forcerange` — **saturated joints mean blocked, not slow**. A skill

## Resolution (same day): pedestal + wrist roll → canary GREEN 10/10

- **[none] THE BIG ONE — position-only IK leaves the wrist ROLL free, and a free roll silently makes grasping impossible.**
  This is the single highest-value lesson of the session and it generalises to every arm+pincer in every simulator.
  Position-only IK (the standard damped-least-squares recipe every tutorial gives you) constrains 3 DOF. On a 5-DOF arm the
  wrist roll is left to whatever the solver drifts into — here, ~0, where the jaws' **pinch axis was 91% VERTICAL**:
  ```
  wrist_roll=0.000  pinch_axis_world=[0.396 -0.133  0.909]   |pinch_z|=0.909   <- jaws close VERTICALLY
  wrist_roll=1.571  pinch_axis_world=[0.224  0.973  0.044]   |pinch_z|=0.044   <- jaws close HORIZONTALLY
  ```
  The gripper was trying to span the cube's **6 cm height** with a **4.2 cm** aperture. Geometrically impossible — and it is
  the complete explanation for a symptom I chased for hours through a dozen unrelated hypotheses (grasp point, approach
  vector, standoff, settle steps, cube size, friction): **the jaws always closed on empty air**. Constraining the roll so the
  pinch axis is horizontal took the oracle from **0/10 → 8/10** in one change.
  **The diagnostic that should be in a skill:** when a gripper closes on air, print the *pinch axis* in world coordinates and
  check it is perpendicular to the object dimension you intend to grasp. Nobody thinks to check this because position IK
  "converged" — residual 1e-5, looks perfect. Orientation is the silent half of the pose.
  **The fix technique:** don't add the roll to the DLS objective — that DIVERGES (tried it: `|pinch_z|` stuck at 0.94, still
  vertical, position error 9 cm). The roll is the last joint before the gripper, so it spins the jaws about the finger axis
  and `pinch_z(roll)` is a smooth 1-D sinusoid → solve it as a **1-D root find**, then re-solve position with the roll
  pinned, and iterate. Converges to `|pinch_z| = 0.002`. Pinning is trivial in MuJoCo: clone the model and set
  `jnt_range[roll] = [r, r]` — the existing `solve_ik` clips to `jnt_range`, so it becomes a roll-constrained solver for free,
  with no second IK implementation.

- **[none] Grasp-offset calibration does NOT transfer across wrist orientations — recalibrate after any orientation change.**
  My throat offset `(-0.02, 0, -0.035)`, empirically calibrated and genuinely correct at the old (roll-free) orientation,
  was worthless once the roll was fixed. Recalibrating under the *actual* grasp orientation gave `(0.015, 0, -0.102)` —
  a completely different point (the fingertip pinch, not the throat wedge). Lesson: the calibration is a function of the
  pose, not a property of the gripper. Re-run it whenever the approach orientation changes.

- **[none] The empirically-calibrated grasp offset is SENSITIVE — sweep it end-to-end, don't eyeball the centroid.**
  Swept through the real oracle: `(0.015,-0.102) -> 9/10`, `(0.015,-0.108) -> 8/10`, `(0.015,-0.114) -> 6/10`,
  `(0.012,-0.108) -> 3/10`. A 3 mm change costs 6/10 episodes. The calibration grid gives you the *neighbourhood*; only an
  end-to-end sweep gives you the point. Same for pedestal height: 0.06 best, and 0.10+ collapses to 0/10 — **higher clearance
  is not monotonically better**, because the arm's reach flattens out.

- **[none] A position servo takes an arbitrary joint-space path — your FIRST waypoint can sweep the arm through the object.**
  Last failing seed (2/10 -> 10/10) was the spawn corner nearest the arm base: the cube was struck at control step 5, during
  the very first move from home, and thrown 0.18 m onto the floor before the gripper did anything. Raising the initial
  approach waypoint (0.16 -> 0.20 m) fixed it. Worth a skill note: waypoint controllers need the first waypoint to be
  reachable *without transiting the workspace*, and "the arm knocked the object over" often means the very first move, not
  the grasp.

- **[none] Scene geometry inherited from a sim asset can be quietly wrong for the robot.** menagerie's scene puts the floor at
  the ARM'S OWN BASE level; a real SO-101 rig has the base and the objects on the same raised work surface. That mismatch —
  not any bug in our code — is what forced the fingers to a ~32 deg down-pitch where the jaw shaft ploughs through the cube.
  A 6 cm pedestal restored the geometry the arm was designed for. **Check that the asset's world matches the robot's intended
  work envelope before assuming your IK/controller is at fault.**

- **[none] Fixed the false-success bug found earlier** (`GRIPPER_GEOMS` name list -> `GRIPPER_BODIES` body membership).
  Confirms the earlier entry: resolving by `model.geom_bodyid` catches the unnamed collision meshes that a name list
  structurally cannot. Verified: 44 tests pass, including a new test pinning the pedestal<->config coupling.

## Wrong/stale guidance — Rerun 0.34.1 API drift

- **[rerun] The brief's Step-1 probe checks the ARCHETYPE names but not the TIMELINE api — and 0.34.1 renamed the timeline
  call too.** The brief warns about `Scalar` → `Scalars` and has a probe for it. That check passed clean:
  `rr.Scalars` (plural) is the correct name in `rerun-sdk==0.34.1`, exact signature
  `Scalars(scalars: datatypes.Float64ArrayLike)` — so a single scalar must be wrapped in a list: `rr.Scalars([float(q)])`,
  not `rr.Scalars(float(q))`. **But the brief's `rr.set_time_sequence("step", step)` does NOT exist in 0.34.1** and the
  probe never covered it. Symptom, verbatim:
  ```
  AttributeError: module 'rerun' has no attribute 'set_time_sequence'
  ```
  Fix: `rr.set_time_sequence` (and `set_time_seconds`/`set_time_nanos`) were unified into a single
  `rr.set_time(timeline, *, sequence=None, duration=None, timestamp=None)`. The sequence form is
  `rr.set_time("step", sequence=step)`. Verified: test_rerun_logger passes and `make viz-oracle` writes a 14.7 MB `.rrd`.
  **Skill upgrade:** robium's `rerun` skill (and this brief's probe) should assert the *timeline* setter alongside the
  archetypes — `hasattr(rr, 'set_time')` vs `hasattr(rr, 'set_time_sequence')` — because the timeline call is on the hot path
  of every logger and drifts on the same release cadence as the archetypes.

## Training pipeline (Task 8)

- **[lerobot] SmolVLA base expects exactly THREE cameras named `observation.images.camera1/2/3`
  — fine-tuning a 2-camera in-embodiment dataset fails hard without a fix, and the free local
  `train-smoke` (5 CPU steps) is what catches it before any GPU spend.** Verbatim error:
  `ValueError: Feature mismatch ... Missing features: ['observation.images.camera1', 'camera2',
  'camera3'] - Extra features: ['observation.images.scene', 'observation.images.wrist']`. The
  fix (LeRobot `rename_map.mdx`, verified via ctx7): `--rename_map={"observation.images.wrist":
  "observation.images.camera1", "observation.images.scene": "observation.images.camera2"}` PLUS
  `--policy.empty_cameras=1` to add a masked placeholder for the missing camera3 (no fake image
  recorded). Supported for SmolVLA/PI0/PI05/PI0Fast/XVLA. **The same rename_map must be applied
  at EVAL/inference too** — the fine-tuned checkpoint keeps the camera1/2/camera-placeholder
  naming, but our env still emits wrist/scene. This is the Task-3 M0 note ("base checkpoint
  declares camera1/2/3, not semantic names") coming due exactly where predicted. Strong `lerobot`
  skill candidate — nothing warns that a semantic-named dataset won't train out of the box.
- **[testing]/[environments] Run the free CPU `train-smoke` (loop-assembles, 5 steps) BEFORE any

## Eval pipeline (Task 9)

- **[lerobot] Closes the open question from Task 8's entry above: at eval, feed the RAW
  (un-renamed) camera keys — the checkpoint's own saved `policy_preprocessor.json` already bakes
  in the rename step from training.** Verified by reading the file directly, not by guessing:
  `outputs/train/smolvla_so101_smoke/checkpoints/000005/pretrained_model/policy_preprocessor.json`'s
  FIRST pipeline step is `rename_observations_processor` with `CAMERA_RENAME_MAP` (wrist->camera1,
  scene->camera2) already in its `config`. Renaming the keys ourselves before calling the
  preprocessor would hand its own rename step keys it no longer recognizes (`observation.images.camera1`
  isn't in a `rename_map` whose *source* keys are `wrist`/`scene`). `evaluate()` still passes
  `CAMERA_RENAME_MAP` explicitly via `preprocessor_overrides` (matching `lerobot_eval.py`'s own
  `predict()`, which re-asserts `cfg.rename_map` even after `pretrained_path=` loading) so the
  rename is sourced from `config.py`, not trusted blindly from the checkpoint JSON — in this case
  both agree. Evidence: `tests/test_smoke.py` passed end-to-end (10 episodes, 139 s) on the first
  attempt with this ordering; no trial-and-error was needed once the JSON was actually read.
- **[lerobot] `SmolVLAPolicy.select_action(batch)` requires a batch that has already been through
  `make_pre_post_processors(...)(raw_batch)` — it does not tokenize/normalize/add-batch-dim
  itself — and the lerobot skill (SKILL.md) does not document this at all for the case that
  actually matters here: a custom, unregistered Gymnasium env.** The skill's entire "Evaluate a
  policy" section is `lerobot-eval --policy.path=... --env.type=<pusht|aloha|libero|...>`, which
  is structurally inapplicable the moment the env isn't one of LeRobot's registered types (true
  for any hand-built sim, not just this one). Found the real contract by reading the installed
  package directly: `lerobot/scripts/lerobot_eval.py`'s `rollout()` (`observation = preprocessor(observation);
  action = policy.select_action(observation); action = postprocessor(action)`) and this repo's
  own `spike/bench_policy.py::_dummy_batch`, written in Task 3, which had already independently
  discovered the same contract for a synthetic batch. **Strong `lerobot` skill candidate**: a
  "write your own rollout loop against a custom env" recipe (load policy -> `make_pre_post_processors`
  from the checkpoint path -> per-step `preprocessor(raw_obs) -> select_action -> postprocessor(action)`)
  is a completely separate path from `lerobot-eval` that the skill doesn't cover, and it's the
  ONLY path available the moment `SO101PickEnv`-style custom envs are in play. Evidence:
  `tests/test_smoke.py` PASSED (139.20 s), `53 passed, 6 deselected` on the full default suite
  afterward — the pattern works, is now shipped in `src/vla_trial/policy/evaluate.py`, and could
  be lifted near-verbatim into a skill example.
- **[lerobot] The postprocessor's un-normalization step is not optional and is easy to silently
  skip.** `policy.select_action()` returns an action in the training-time normalized (MEAN_STD)
  space, not actuator units — `policy_postprocessor.json`'s first step is `unnormalizer_processor`.
  Caught this by reading the postprocessor JSON before writing the rollout loop (not by a failed
  rollout producing garbage motion); `evaluate()` runs `postprocessor(action)` unconditionally
  before `env.step()`. Same class of gotcha as the camera rename above — the checkpoint's own
  saved processor JSON is the ground truth, worth checking before assuming pass-through.

- **[lerobot]/[huggingface] `--output_dir` is passed VERBATIM to the remote HF Jobs container —
  a local absolute path makes the job train to completion and THEN crash at checkpoint save.**
  Cost a full ~$2-4 pipe-test run for zero artifact. Symptom: 2000/2000 steps done, then at
  `save_checkpoint -> policy.save_pretrained(output_dir)` the remote Linux box tries to
  `os.mkdir` the Mac path and dies with `PermissionError: [Errno 13] Permission denied: '/Users'`
  (recursing up the nonexistent `/Users/jazarium/repos/...`). Because save happens BEFORE the
  Hub push, `--policy.push_to_hub=true` never fires — no checkpoint anywhere, job stage=ERROR.
  Fix: remote runs must set `--output_dir` to a container path (`/tmp/...`), NOT a local
  APP_ROOT-derived one; the Hub-pushed repo is the real artifact so the remote dir is just
  scratch. Regression guard: assert the remote command contains no `~`/`/Users/` path.
  General rule worth a skill line next to any `--job.target` example: **every path argument to a
  remote job must be container-valid; local absolute paths silently pass submission and fail at
  the first filesystem write (often the very end).** Also: default `save_freq=20000` means a
  2000-step run's ONLY save is at the end, so this bug surfaces exactly when the run "finished".

- **[lerobot]/[huggingface] On the HF Jobs path, lerobot-train IGNORES `--policy.repo_id` for the
  final model push — it publishes to an auto-generated `<user>/train_<timestamp>` repo instead.**
  We passed `--policy.repo_id=jazarium/smolvla_so101_pick`; the model landed at
  `jazarium/train_2026-07-15_08-09-36` and `smolvla_so101_pick` was never created. So after a
  remote run you must read the "Model pushed to <url>" line to learn the real repo id (or set
  `--job_name` to control it) — you cannot assume the checkpoint is at policy.repo_id. Eval must
  target the actual pushed repo. Worth a skill note next to any remote-train example.
- **[lerobot] Eval auto-switches device cuda->mps when no CUDA is present** (`Device 'cuda' is
  not available. Switching to 'mps'`), so a checkpoint whose train_config says cuda evals fine on
  a Mac with no code change — confirms the M0 local-MPS inference path (0.55 s/pass) for the real
