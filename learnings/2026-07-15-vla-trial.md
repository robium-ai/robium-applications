# 2026-07-15 — vla-trial, Task 10 (narrative harness) + Task 11 (closeout)

## Figured out from scratch — a regression test silently went stale when a config constant changed

- **[testing] A hardcoded literal in an assertion (`"--steps=2000" in pipe`) survived a config change
  that made it structurally false, and nothing caught it until the next full-suite run.** Task 8's
  `test_pipe_test_and_full_differ_in_steps` (tests/test_train_starts.py) asserted the exact string
  `--steps=2000`, matching `PIPE_TEST_STEPS` at the time it was written. The 2026-07-15 user directive
  ("have 100 iterations only") dropped `config.PIPE_TEST_STEPS` 2000 -> 100 to minimize spend on the
  pipe-test run (see config.py) — the command builder (`train_remote_cmd`) correctly picked up the new
  value, but the test's hardcoded string didn't, so it started asserting a fact that was no longer true.
  Verbatim failure caught while re-verifying `make test` for Task 11 closeout, days after the config
  change actually landed:
  ```
  assert '--steps=2000' in 'lerobot-train ... --steps=100 --batch_size=32 ...'
  ```
  Fix: import `PIPE_TEST_STEPS`/`TRAIN_STEPS` from `config.py` and assert `f"--steps={PIPE_TEST_STEPS}"`
  instead of a literal — the same "single source of truth" rule this repo already applies to Makefile
  targets should apply to test assertions of config-derived command strings, not just to the config
  values themselves. **General rule worth a `testing` skill line: when a regression test asserts an
  exact rendered value (a CLI string, a rendered path, a formatted number) that is *derived* from a
  config constant, assert against the constant, never re-type the literal — otherwise the test doesn't
  regress with the code, it silently starts testing a stale fact.** Evidence: `make test` was
  `54 passed, 7 deselected` after the fix (was `53 passed, 1 failed, 7 deselected` before).

## Figured out from scratch — the raw base checkpoint isn't eval-compatible with our own fine-tuned ones

- **[lerobot] `evaluate()`'s `empty_cameras` guard rejects the untouched base checkpoint outright —
  writing the Task 10 narrative harness against `BASE_POLICY_ID` directly fails before a single
  episode runs.** Our fine-tuned checkpoints are trained with `--policy.empty_cameras=1` (a masked
  placeholder for the missing 3rd camera — see the 2026-07-14 learnings' camera-rename entry), so
  `evaluate()` asserts `policy.config.empty_cameras == EMPTY_CAMERAS(1)` as a footgun guard against
  eval-ing a checkpoint trained under a different camera layout. The raw `lerobot/smolvla_base` was
  never fine-tuned under that layout — its own `config.empty_cameras` is `0` (it expects 3 REAL
  cameras) — so loading it straight into `evaluate()` trips that exact assertion. Verified directly,
  not guessed:
  ```python
  SmolVLAPolicy.from_pretrained("lerobot/smolvla_base").config.empty_cameras  # -> 0
  ```
  vs. both of our own checkpoints (5-step local smoke, 100-step Hub pipe-test), which read `1`. Fix
  for the harness test (`tests/test_narrative.py`): use two checkpoints that ARE eval-compatible — the
  5-step local smoke checkpoint and the 100-step Hub pipe-test checkpoint (`jazarium/train_2026-07-15_
  08-09-36`, already HF-cached from Task 9) — genuinely two different, real, trained artifacts, just
  not "base vs. fine-tuned" in the demo's eventual sense. **This is a real gap for the eventual full
  narrative script, not just a test-writing inconvenience**: comparing against the TRUE untouched base
  model will need its own eval path (3 real camera keys, no rename/empty_cameras override) alongside
  the fine-tuned path, or a documented compatibility shim. Filed as the harness test's grep-able
  `TODO(full-training)` rather than worked around silently. Candidate `lerobot` skill line: "a
  fine-tuned checkpoint's processor config (camera count, rename map) is NOT interchangeable with the
  base checkpoint's own — evaluating both through one code path needs an explicit compatibility check,
  not an assumption that `from_pretrained` + your training-time overrides always apply."
