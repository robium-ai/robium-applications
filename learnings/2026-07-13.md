## Figured out from scratch — VLA model selection

- **[lerobot] / [none] no VLA model-selection guidance exists in any skill, and the choice is
  nearly forced once the numbers are on the table.** Compiled by hand:

  | Model | Params | Runs without a GPU? | Evidence |
  | --- | --- | --- | --- |
  | SmolVLA (`lerobot/smolvla_base`) | 450M | **Yes** (HF: "CPU… or even a MacBook") | HF blog, arXiv 2506.01844 |
  | pi0 / pi0-FAST (openpi) | ~3B | No — >8 GB VRAM; openpi repo is **Ubuntu-only** | openpi README |
  | MolmoAct2 | 5.5B | No (see below) | model card |
  | OpenVLA | 7B | No — ~14 GB VRAM | — |
  | GR00T N1.5/N1.7 | 2–3B | No — needs flash-attn + CUDA 12.4 | NVIDIA/Isaac-GR00T |

  SmolVLA is the only member of the "runs on a laptop" class. Its pretraining corpus is SO-100
  data *exclusively* (487 community datasets, ~10M frames) — so an SO-100/101 fine-tune is
  **in-embodiment**. Consequence worth stating in the skill: **the arm choice and the model
  choice are one choice**, not two.
- **[none] "zero-shot" on a VLA model card means zero-shot on the *embodiment*, not on your
  *renderer*.** Cost ~an hour. `allenai/MolmoAct2-SO100_101` is real (5.4k downloads) and
  genuinely zero-shot on SO-100/101 — but (1) it is **5.49B params, CUDA-only in practice**
  (every documented path is `.to("cuda")` with `enable_cuda_graph=True` *by default*; ~26 GB
  fp32 / "under 16 GB" bf16), and (2) it was trained on **RealSense images of real scenes**
  (sample inputs are literally `sample_realsense_top_rgb.png`), so feeding it MuJoCo renders is
  a real→sim domain gap in the unfavourable direction. Not falsified by a run (we declined to
  rent a GPU) — **tentative** — but the memory numbers alone settle it for a laptop demo.

## Figured out from scratch — MuJoCo on macOS (feeds the unwritten `mujoco` skill)

- **`MUJOCO_GL=cgl` is *the* macOS headless-render backend.** `osmesa` and `egl` are
  **Linux-only** (mujoco#2164 is the open port request). Without this there is no obvious way to
  render policy camera observations without a display on a Mac — a hard blocker, silently.
- **The interactive viewer (`viewer.launch_passive`) needs `mjpython`, not `python`,** on macOS
  (main-thread rendering constraint); also a known bug under `uv` (mujoco#1923). Guaranteed
  first-run stumble.
- **Version-pin trap:** `gym-hil`/`gym-aloha` pin `mujoco<3.9`, `gym-xarm` pins `<3.0`, but
  `mujoco_menagerie`'s SO-101 needs **≥3.1.3** and current MuJoCo is **3.10**. A hard dep on
  `gym-hil` silently *downgrades* MuJoCo below the model's own requirement. Copy its patterns;
  don't depend on it.
- **Offscreen render throughput on M-series is unmeasured** (upstream: "Offscreen rendering with
  mjr_render is extremely slow"). M0 spike, not yet answered.

## Figured out from scratch — training economics

- **[lerobot] MPS fine-tuning is a non-starter, and the skills say nothing about it.** LeRobot
  *supports* `--policy.device=mps`, which actively invites you to try. Reported data point:
  **~2 hours to complete 20 steps.** A fine-tune is 20,000. MPS is for inference and for proving
  the loop starts — never for training. Deserves a loud line in `lerobot` and/or `environments`.
- **[lerobot] LeRobot 0.6.0 made remote training one flag:** `lerobot-train
  --job.target=a10g-small …` pushes the dataset to the Hub if needed, submits to HF Jobs, and
  streams logs back to the terminal. That is the entire local-Mac / remote-GPU story with **zero
  infra to build** — exactly what `environments` claims to care about. Costs: **20k steps ≈ 4 h
  on an A100** (HF docs); an independent practitioner run (75 eps, 20k steps, batch 64) took
  **~10.4 h on a 3090 → 60–80% success**. So ~$5–15 per fine-tune.
- **[data] / [none] episode *density* beats episode *count*, and the failure boundary is
  documented but obscure.** 25 eps → bad. **50 eps over a 30 cm workspace → outright failure**
  ("learned the general motion but couldn't pin down grasp locations"). **75 eps over ~10 cm →
  80% success.** Anyone planning "~50 episodes" — the number that feels natural, and the number
  I first proposed — is sitting exactly on the wrong side of the boundary. Candidate: a dataset
  sizing rule in `data` or `lerobot`.
- **LeRobot 0.6.0 requires Python ≥3.12** — silently breaks 3.10/3.11 envs.

## Figured out from scratch — UI / viz

- **[huggingface] Gradio has NO HuggingFace dependency, and users assume it does.** It is a plain
  Python web app (FastAPI + uvicorn, :7860). Three self-hosting mechanics worth one short skill

# 2026-07-13 — vla-trial Task 3 (M0 policy spike — build block)

- **[lerobot] figured out from scratch: `select_action()` does NOT tokenize the task string — the batch must be run through the policy's own pre-processor pipeline first.** The natural reading of the API (`policy.select_action(batch)` with `batch["task"] = "put the cube in the bin"`) fails with `KeyError: 'observation.language.tokens'` at `modeling_smolvla.py:290`. The tokenization/normalization/batching live in a *separate* pipeline built by `make_pre_post_processors(policy_cfg=..., pretrained_path=...)` (`lerobot.policies.factory`, re-exported from `lerobot.policies`), which you call on the raw obs dict before `select_action`. Verified: with the preprocessor, a forward pass returns a `(1, 6)` action tensor; without it, `KeyError` every time. Dead end ruled out: passing `task` as a list vs a string changes nothing — the key is missing either way, it is not a shape bug. Gotcha #2 in the same call: `make_pre_post_processors` reads `device` from the *checkpoint's* config, which is `cuda`; on a Mac it raises `AssertionError` inside `get_safe_torch_device` unless you pass `preprocessor_overrides={"device_processor": {"device": device}}`. Both are invisible in the model card. High-value for the `lerobot` skill.
- **[lerobot] the pretrained `smolvla_base` checkpoint declares GENERIC camera keys, not your env's.** `policy.config.image_features` is `observation.images.camera1/2/3` (each `(3,256,256)`), NOT `observation.images.wrist`/`.scene`. Passing your own names raises `ValueError: All image features are missing from the batch. At least one expected.` — a confusing message, since the batch is full of images; it means *none of the names matched*. Build benchmark batches from `policy.config.image_features` / `.robot_state_feature` rather than hardcoding, and treat env→policy camera remapping as an explicit step (LeRobot has a `rename_map` / `RenameObservationsProcessorStep` for exactly this).
- **[lerobot]/[environments] MEASURED: SmolVLA forward-pass latency on Apple Silicon (M-series), 450M `smolvla_base`, 20 timed passes, 5 warm-up, `policy.reset()` before every timed call.** No published benchmark for this existed anywhere; HF's "runs on a MacBook" is a marketing claim. **MPS 0.536 s mean / 0.543 p95. CPU native 9.186 s / 9.399. CPU in a linux/arm64 container 9.318 s / 9.621.** Two conclusions the skills should carry: (1) **MPS is 17x faster than CPU for VLA inference** — "runs on a MacBook" is true *only* with MPS; on CPU it is ~9 s/chunk, which is not a demo, it is a slideshow. (2) **Containerising costs ~nothing (+1.4%) — the penalty is losing MPS, not the container.** Since Docker-on-macOS cannot see MPS and Cloud Run has no GPU, any containerized/deployed VLA demo is on the 9 s path. This *confirms and quantifies* the existing "(seen 2x)" note that Docker-on-macOS loses MPS and that CPU is the number that decides the architecture. **(seen 3x)**
- **[lerobot] action chunking makes a naive benchmark report a fantasy number, and you have to know to defend against it.** SmolVLA's `select_action` serves ~50 subsequent calls from a cached chunk (`self._queues[ACTION].popleft()`), so timing it in a loop without `policy.reset()` measures a deque pop, not a forward pass. Evidence the guard works: with `reset()` every timed call costs ~9.2 s on CPU, and the spread is tight (8.99–9.65 over 20 passes) — real compute, not a cache. The trap is that the un-guarded version *looks* like a spectacular result rather than an error. Worth a warning in the `lerobot` skill and in `testing` (benchmarking a cached API).
- **[testing] a benchmark harness that swallows exceptions can report success while measuring nothing.** The dispatcher pattern `for device in devices: try: bench(device) except Exception as e: print(f"{device}: unavailable ({e})")` then `return 0` — which is what the task brief prescribed — let `make spike-policy-container` exit **0** while producing **zero** measurements (the container hit a transient HF Hub `401 Unauthorized` from `cas-server.xethub.hf.co` fetching weights). A measurement command must fail loudly when it measured nothing; fixed by counting successes and returning 1 if none. General rule for `testing`: **for a benchmark, "no data" is a failure, not a quiet skip.**
- **[testing] a spike that writes results keyed by device silently self-clobbers when the same device is measured in two runtimes.** `outputs/spike/policy.json` keyed on `"cpu"`, plus a container that bind-mounts the same `outputs/`, meant the container-CPU run **overwrote the native-CPU run** — destroying exactly the two numbers the spike existed to compare. Caught only because the native number was still in a scratch log. Key results by *(device, runtime)*, not device.
- **[environments] `docker build` of a torch-based image on linux/arm64 pulls the full NVIDIA CUDA wheel set** (`nvidia-cudnn-cu13`, `nvidia-nccl-cu13`, `triton`, …) even though the image is CPU-only and arm64 — a 3.46 GB image and a multi-minute build for a benchmark that never touches a GPU. Installing the CPU-only torch index (`--index-url https://download.pytorch.org/whl/cpu`) would cut this substantially. Not fixed here (the fat image measures the same thing), but any real deploy of this container should. Candidate: an `environments` note on torch-in-Docker image bloat.
- **[none] container re-runs re-download ~1 GB of weights every time unless the HF cache is mounted.** `-v $HOME/.cache/huggingface:/root/.cache/huggingface` makes container spike runs fast and repeatable — and it is what stopped the transient HF 401 from recurring. Obvious in hindsight; cost two failed runs.
- **[testing] the same clobbering bug bit a THIRD time, from the test suite — `make test` silently destroyed the M0 benchmark. (seen 3x in one task)** The brief's test called `bench_policy(device="cpu", n_passes=3)` with no output redirection, so a throwaway 3-pass smoke run wrote straight into the canonical `outputs/spike/policy.json` and **overwrote the real 20-pass benchmark**. Caught only by re-reading the artifact after commit and noticing `n=3` where `n=20` should be. Generalizable rule worth putting in `testing`: **a test must never write to a real output artifact.** Any bench/measure function needs an injectable output path, the test passes `tmp_path`, and — the part that actually holds the line — a regression test asserts the canonical artifact is byte-identical before and after a test run. Three variants of "shared mutable output path" in one task (container-vs-native, test-vs-real) says this is a systemic design smell in spike harnesses, not bad luck.
