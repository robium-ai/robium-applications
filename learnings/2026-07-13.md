
- [live-demo] figured out from scratch (demo orchestrator build): the in-container gateway can't manage its own lifecycle (it dies with the container it would need to restart), and a browser can't start containers. The missing piece locally is a host-level **orchestrator** — a small always-on service (Node/TS + dockerode) that owns start/stop/list/limit across demos and hands the browser a per-instance connect host; the browser then talks to that instance's gateway directly (lifecycle-only, never in the data path). This is the exact role Cloud Run's control plane plays in prod, and a clean `Driver` interface lets LocalDockerDriver (now) and a future CloudRunDriver share the API. Verified end-to-end: page Start spawns a fresh container on an ephemeral port → sim ready rtf 1.2 → Stop removes it, zero manual `make demo`. Strong candidate for a live-demo skill architecture reference (the gateway-vs-orchestrator split + the local/cloud driver seam).

# 2026-07-13 — demo v4 IDE workspace + orchestrator (gotchas)

- [live-demo] figured out from scratch (IDE Console pane): a real in-browser terminal into a demo container = xterm.js on the client + a **PTY over WebSocket** on the gateway. No ws library needed inside the container — hand-roll the RFC6455 frame codec (mask/unmask, opcodes) in ~40 lines of stdlib; `pty.fork()` + `os.execvp('bash')` + pipe bytes both ways. Bridge subprotocol note still applies (foxglove.sdk.v1). Candidate content: live-demo skill's IDE/terminal reference.
- [live-demo] security shape for a public shell (deployed but verification deferred): a real shell into a Cloud Run container is defensible ONLY with (1) a dedicated **zero-IAM-role service account** (a stolen metadata token then unlocks nothing — which forced moving the fleet-count query off the demo container) and (2) **VPC deny-all egress** (Direct VPC egress + a deny-all firewall so the container can't dial out; the sim needs no egress — all traffic is the inbound visitor ws). Cloud Run gen2 microVM isolation handles container-escape; these two handle the actual holes (creds + egress). See robium.org docs/BACKLOG.md for the pending egress-block E2E verification.
- [live-demo] Cloud Run Direct VPC egress needs a subnet big enough for the instance pool: a `/28` (16 IPs) failed deploy with "no sufficient IP addresses in the VPC network" at max-instances=5 + 8 vCPU; a `/24` fixed it. `gcloud compute networks subnets expand-ip-range` widens in place.
- [live-demo]/[integration] CONCURRENT sims must not share a ROS_DOMAIN_ID. The moment an orchestrator makes >1 sim run at once, hardcoding one domain (42) makes their ROS graphs merge over the Docker network → two Gazebo `/clock` publishers → `[robot_state_publisher] Moved backwards in time, re-publishing joint transforms!` floods forever. Fix: assign the lowest-free ROS_DOMAIN_ID (1..200) per instance in the driver, label it, exclude used ones. (gz-transport side is already isolated by GZ_RELAY=127.0.0.1 = loopback-only, per-container.) Candidate: live-demo skill + a note in the orchestrator pattern.
- [live-demo] frontend gotcha with dynamic backends: once Start picks a per-instance host (ephemeral port), a `setInterval(poll)` closure captures the STALE host and polls the wrong backend forever (symptom: logs stream fine via a re-rendered child, but the ready-state/Foxglove-button never flips). Fix: poll reads a `hostRef.current`, not the `host` closed over at interval-creation. General lesson for any UI whose backend address changes mid-session.
- [live-demo] local-dev ergonomics that every demo will want: (1) a `?host=`/switcher to point a HMR frontend at prod-cloud OR a local backend; (2) same-site subdomain (demo.<domain>) + credentialed fetch + localhost-CORS so Cloud Run's affinity cookie actually routes; (3) cloud-disposable Stop/close semantics are WRONG for a warm local container — but the real fix is the orchestrator (Stop truly removes, Start truly spawns, everywhere) rather than special-casing localhost; (4) bind `npm run dev` to launch site + orchestrator together (concurrently). These are the difference between a demo you can iterate on and one you fight.
